입금 완료 처리는 최고관리자만 가능하다.

04_FIREBASE_DECISION_MATRIX - 기존 Firebase vs 신규 Firebase 판단
공식 기준
Firebase는 개발 워크플로에서 환경별 별도 Firebase 프로젝트 사용을 권장한다. 즉 개발(dev), 테스트/staging, 운영(prod)을 같은 프로젝트에 섞는 구조는 권장하지 않는다.
결론 권장안
이 프로젝트는 결제, 비회원 주문정보, QR 세션, 입점사 정산, 알림톡, 외부 API가 들어가므로 신규 Firebase 프로젝트를 최소 2개 이상 만드는 것을 기본 권장한다.

기존 Firebase를 써도 되는 조건
기존 Firebase 프로젝트는 아래 조건을 모두 만족할 때만 사용한다.
대표님이 Owner 또는 충분한 IAM 권한을 갖고 있다.
운영 고객 데이터가 없다.
기존 앱/서비스가 운영 중이 아니다.
Firestore Rules를 재설계해도 영향받는 사용자가 없다.
Billing, IAM, API Key, Functions, Hosting 상태가 명확하다.
기존 .firebaserc와 firebase.json이 정리되어 있다.
외부 연동 키/Secret이 코드에 노출되어 있지 않다.
dev 용도로만 사용할 수 있다.
신규 Firebase가 필요한 조건
아래 중 하나라도 해당하면 신규 Firebase를 만든다.
운영 고객 데이터가 있다.
프로젝트 소유자/권한이 불명확하다.
기존 앱이 이미 연결되어 있다.
Firestore 구조가 기존 기능과 충돌한다.
Secret Key 관리가 불명확하다.
개발/운영 환경이 같은 프로젝트에 섞여 있다.
PG/알림톡/외부 API 실험을 해야 한다.
Codex 자동화가 개발 프로젝트에서 테스트를 반복해야 한다.
Codex의 Firebase 감사 명령어
현재 프로젝트의 Firebase 설정을 감사해라. 코딩하지 마라. 확인: 1. .firebaserc 존재 여부 2. firebase.json 존재 여부 3. firestore.rules 존재 여부 4. firestore.indexes.json 존재 여부 5. functions 디렉터리 존재 여부 6. Firebase app config 존재 여부 7. service account/private key 파일 존재 여부 8. .env 파일 존재 여부 9. dev/staging/prod 분리 여부 10. 기존 Firebase를 재사용할 수 있는지 판단 결과를 FIREBASE_DECISION_REPORT.md로 작성해라.

현재 컴퓨터/프로젝트에서 설치 및 설정 상태를 점검해라. 코딩하지 마라. 점검: Git, GitHub remote, Node.js, pnpm/npm, Next.js, TypeScript, Firebase CLI, Google Cloud CLI, Flutter SDK, Android Studio, Docker, Postman, Codex CLI, AGENTS.md, Firebase 설정, .firebaserc, firebase.json, functions, Cloud Run 구조. 결과를 INSTALLATION_STATUS_REPORT.md로 작성하고, 설치 우선순위를 1순위부터 제시해라.
3. Firebase 결정 명령어
현재 A 프로젝트의 Firebase 설정을 분석해서 기존 Firebase를 사용할지, 새 Firebase 프로젝트를 만들지 판단해라. 코딩하지 마라. 확인: .firebaserc, firebase.json, firestore.rules, storage.rules, functions, app config, service account 파일, .env, dev/prod 분리 여부, 운영 데이터 위험. 결과를 FIREBASE_DECISION_REPORT.md로 작성하라. 최종 판단은 USE_EXISTING_FOR_DEV_ONLY, CREATE_NEW_DEV_AND_PROD, CREATE_NEW_DEV_STAGING_PROD, BLOCKED_NEED_OWNER_PERMISSION, BLOCKED_SECRETS_EXPOSED 중 하나로 표시하라.
4. Gap Analysis 명령어
본 문서팩 요구사항과 현재 A 프로젝트 파일을 비교해라. 코딩하지 마라. 비교 대상: 최고관리자, 기업 Admin, 산후조리원 Admin, 객실/태블릿, 태블릿 폐쇄몰, QR 결제, PG, 알림톡, 배송조회, 외부 재고 API, 정산/입금, 보안/로그. 각 항목을 Complete, Partial, Missing, Risk, Unknown으로 분류하고, Keep, Modify, Replace, Create, Block 중 하나의 조치를 제안해라.
5. 승인 후 1차 생성 명령어
대표님이 A_PROJECT_GAP_ANALYSIS.md와 FIREBASE_DECISION_REPORT.md를 승인했다. 이제 1차 생성 작업을 시작한다. 범위: 문서/폴더 구조/관리자 기본 라우팅만 생성한다. 금지: PG, 환불, 정산, Firestore 운영 배포, Secret Key, 운영 DB 접근. 완료 후 AUTO_REPORT.md를 작성해라.
6. 퇴근용 명령어
SAFE_NIGHT_MODE로 작업한다. 위험도 낮은 A등급 작업 1개만 수행한다. 테스트 2회 실패, 수정 파일 15개 초과, 결제/환불/정산/보안 파일 수정 필요 시 즉시 중단하고 BLOCKERS.md에 기록한다.
7. 외근용 명령어
SAFE_FIELD_MODE로 작업한다. 코드 수정하지 말고 오류 분석, 작업 범위 제안, 다음 Codex 명령어만 작성한다. 결제/환불/정산/보안/운영 배포 관련 수정은 금지한다.
