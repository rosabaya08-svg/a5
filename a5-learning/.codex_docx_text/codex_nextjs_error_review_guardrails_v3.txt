문서 목적
이 문서는 기존 Codex × Next.js 문서와 자동화 마스터 문서를 재검토하여, 오류 가능성이 있는 표현을 정정하고 자동 코딩의 작용 영역과 부작용 영역을 분리한 실행 가드레일 문서입니다. 핵심은 Codex가 요청 범위 안에서만 코딩하고, 범위 밖 개선은 제안으로만 남기며, 위험 영역은 BLOCKERS.md로 중단하게 만드는 것입니다.

1. 재검토 결과 요약
확인됨: 업로드된 자동화 마스터 문서는 자동화를 운영 시스템 무인 완성이 아니라 개발 브랜치 또는 worktree 안에서 안전한 기능을 생성·테스트·오류 검토·보고하는 구조로 정의합니다.
확인됨: 업로드된 쇼핑몰 설계 문서는 QR 세션, 결제, 정산, 권한, Secret Key, 외부 API, 알림톡, 배송조회가 상용 커머스의 사고 위험 영역임을 전제로 합니다.
재검토 결론: 기존 문서의 방향은 맞지만, “완전자동”, “오류 수정”, “Browser/Computer Use”, “B등급 초안” 표현은 운영자가 넓게 해석하면 위험합니다. 이번 문서에서는 자동화 허용 범위를 더 좁히고, C등급 중단 규칙을 우선 적용하도록 정정했습니다.
최종 운영 원칙
자동 코딩은 허용하되 자동 결정은 금지합니다. Codex는 작성·수정·테스트·보고까지, 대표님은 방향 결정·운영 승인·금전 처리·정책 확정까지 담당합니다.

6.3 오류 수정 제한
오류 수정은 무제한 반복하지 않습니다. 자동 수정은 안전 영역에서 최대 2회까지로 제한하고, 결제·환불·정산·보안·운영 DB·Secret·배포 파일에 접근해야 하면 중단합니다.
오류 수정 규칙: - 임시 우회 금지 - 기존 기능 삭제 금지 - 테스트 약화 금지 - any 남발 금지 - 수정 파일 최소화 - 최대 2회 자동 수정 - 2회 실패 시 BLOCKERS.md 기록 - 위험 영역 접근 시 즉시 중단

MCP
Next.js 16+ 및 next-devtools-mcp 설치 조건 확인 후 build/runtime/type error 확인에 사용
7.3 Firestore Rules 보완
중요 보완
Firestore Security Rules는 모바일/웹 client SDK 요청을 통제하는 장치입니다. 서버 client libraries, Cloud Functions, Cloud Run에서 발생하는 서버 측 접근은 Rules만으로 통제된다고 보면 안 됩니다. 서버 코드 권한검증, IAM, Secret Manager, App Check, audit_logs를 함께 설계해야 합니다.
접근 위치

웹/모바일 client SDK
Firebase Auth + Security Rules + App Check + Rules unit test
Server Action / Route Handler
세션/claim 검증, role/company_id/nursery_id/tablet_id 검증, 서버 재계산
Cloud Functions / Cloud Run
IAM, Secret Manager, 서버 입력 검증, audit_logs, idempotency

SUPER_ADMIN 권한, 이중 확인, 금액/상태 변경 감사 로그

10. 바로 쓰는 Codex 프롬프트
10.1 오류 재검토 시작 프롬프트
오늘 작업 전 기존 분석 내용을 재검토한다. 먼저 읽을 문서: - AGENTS.md - PROJECT_RULES.md - DB_SCHEMA.md - STATUS_MODEL.md - SECURITY_POLICY.md - AUTO_MODE_POLICY.md - BLOCKERS.md - NEXT_TASKS.md - AUTO_REPORT.md 출력: 1. 기존 분석 중 오류 가능성이 있는 부분 2. 자동화 등급 재분류가 필요한 부분 3. 방향 이탈 위험이 있는 부분 4. 거짓 코딩/외부 API 추측 위험 5. 오늘 코드 수정 가능 여부 아직 코드는 수정하지 마라.
10.2 방향 이탈 방지 프롬프트
요청 범위: [구체적 기능 1개] 수정 가능: [파일/폴더] 수정 금지: [파일/폴더/기능] 규칙: - 요청 범위 밖은 코드로 구현하지 마라. - 더 좋은 개선안이 보이면 NEXT_TASKS.md에 제안만 남겨라. - 위험 영역이 필요하면 즉시 중단하고 BLOCKERS.md에 기록하라. - 수정 전 계획을 먼저 제시하라.
10.3 거짓 코딩 방지 프롬프트
공식 문서 또는 프로젝트 문서로 확인되지 않은 API, endpoint, parameter, response format, Secret Key 이름을 만들지 마라. 확인되지 않은 연동은 mock adapter로만 작성하라. 실제 연동이 필요하면 필요한 공식 문서, 테스트키, 계약 정보, 승인 절차를 BLOCKERS.md에 작성하라. 테스트를 실행하지 않았다면 성공했다고 쓰지 마라. 실패한 테스트는 실패했다고 기록하라.
10.4 오류 수정 프롬프트
오류를 수정해라. 원칙: - 임시 우회 금지 - 기존 기능 삭제 금지 - 테스트 약화 금지 - any 남발 금지 - 수정 파일 최소화 - 최대 2회까지만 자동 수정 - 결제/환불/정산/보안/운영 배포/Secret Key/운영 DB 파일 수정 필요 시 중단 출력: 1. 오류 원인 2. 관련 파일 3. 안전한 수정 가능 여부 4. 수정 계획 5. 수정 내용 6. 실행한 테스트 7. 실패한 테스트 8. 사람 확인 필요 항목
10.5 자동화 금지 중단 프롬프트
중단한다. 이 작업은 자동화 금지 영역이다. 아래에 해당하면 코드 수정하지 마라: - PG 운영키 - 실결제 - 운영 환불/부분취소 - 정산 지급 - 운영 DB 삭제/마이그레이션 - Secret Key 변경 - 운영 배포 - 개인정보/약관/법무 판단 BLOCKERS.md에 필요한 공식 문서, 키, 계약 정보, 승인 절차, 다음 조치만 작성하라.
